你刚刚完成了一项任务。现在不要急着庆祝，先以审计者和系统优化工程师的视角质疑这次执行：

**这次任务为什么花了这么久？中间到底走了哪些弯路？问题究竟来自系统提示词、任务规划、模型推理、工具能力、工具调用策略、上下文管理，还是外部环境？**

请基于本次任务的实际执行过程回答，不要泛泛而谈，不要为了显得专业而编造原因。所有结论必须区分为：

* **已证实** ：能由任务日志、工具调用、报错、返回结果或最终产物直接支持。
* **高概率推断** ：有合理迹象，但缺少直接证据。
* **未知 / 需要补充观测** ：当前无法可靠判断，说明应增加什么日志、指标或实验。

请按以下结构输出：

# 1. 执行结果审计

* 任务目标是什么？
* 最终是否完成？完成质量如何？
* 哪些产出真正推动了任务完成，哪些产出价值有限？
* 是否存在返工、重复搜索、重复读取、重复计算、无效工具调用或不必要的长推理？

# 2. 时间与路径复盘

按执行顺序列出关键步骤，并标记：

* 步骤目的
* 使用的模型能力或工具
* 实际收益
* 时间 / 轮次成本
* 是否必要
* 是否可以合并、提前终止、缓存、并行或跳过

重点指出最昂贵的 1～3 个环节，以及它们为什么昂贵。

# 3. 弯路与根因分析

逐项分析本次任务的低效点。每项必须包含：

* 现象
* 直接原因
* 更深层根因
* 证据等级：已证实 / 高概率推断 / 未知
* 影响：耗时、成本、成功率、质量、用户体验
* 可执行修复方案

请至少从以下维度检查，不适用时明确说明原因：

1. 系统提示词是否过长、冲突、重复、限制过多，或缺少决策边界？
2. 任务拆解是否过细、过度规划，或没有尽早开始有效执行？
3. 是否缺少“何时停止搜索 / 停止推理 / 停止调用工具”的机制？
4. 工具能力是否不足，还是工具选择、参数、调用顺序不佳？
5. 是否存在工具返回信息不足，导致重复调用或模型猜测？
6. 上下文是否过载、遗漏、重复注入，或缺少结构化状态管理？
7. 是否应该使用缓存、摘要、索引、批量调用、并行调用或持久化中间结果？
8. 是否存在模型不确定却没有主动验证，或已经有证据却继续无效验证的情况？
9. 是否存在不必要的澄清、重复确认、格式化、解释或面向用户的过程性输出？
10. 是否是外部依赖、网络、权限、文件体积、API 限流或运行环境造成的瓶颈？

# 4. 归因结论

请对以下来源给出影响权重，总和为 100%，并解释依据：

* 系统提示词设计：__%
* Agent 编排 / 任务规划逻辑：__%
* 模型能力或推理策略：__%
* 工具能力缺口：__%
* 工具调用策略与参数设计：__%
* 上下文 / 记忆 / 状态管理：__%
* 外部环境与不可控因素：__%

不要把问题笼统归咎于“模型不够聪明”或“工具不够强”。必须继续追问：**具体缺什么能力、在哪个决策点缺失、补上后会减少哪一步或提升什么指标。**

# 5. 面向源码的优化建议

按优先级输出一个优化 Backlog。每项包含：

* 优先级：P0 / P1 / P2
* 改动位置：系统提示词、Planner、Executor、Tool Router、Memory、Observability、Cache、UI 或其他模块
* 建议改动
* 预期收益
* 实现复杂度：低 / 中 / 高
* 风险或副作用
* 验证指标与 A/B 实验设计

优先提出能直接在源码中实现的改动，例如：

* 精简或重构系统提示词；
* 增加任务复杂度分类与不同执行模式；
* 设置工具调用预算、重试预算、搜索停止条件和推理停止条件；
* 设计工具选择路由规则；
* 增加批量调用、缓存、结果去重和并行策略；
* 为工具返回增加结构化字段、置信度、耗时和失败原因；
* 引入状态机、检查点、摘要压缩和上下文淘汰机制；
* 增加失败归因日志、阶段耗时、token 消耗、工具成功率与返工率指标。

# 6. 最小可行改进方案

最后只选择  **1 个最值得先做的改动** ，说明：

* 为什么它比其他改动更优先；
* 最小实现方案；
* 需要修改的核心逻辑；
* 如何判断它确实让系统更快、更便宜或更可靠；
* 若实验失败，下一步应验证什么假设。

## 输出原则

* 直言不讳，允许批评系统设计。
* 不要给空泛建议，例如“优化提示词”“增强工具能力”“提升模型推理”。
* 每条建议必须落到具体机制、代码模块、控制条件或可观测指标。
* 若本次执行已经接近最优，也要说明依据，以及还缺失哪些观测数据来证明这一点。
