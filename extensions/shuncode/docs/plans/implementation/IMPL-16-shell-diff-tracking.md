# IMPL-16: Отслеживание изменений файлов из Shell (External Diff Tracking)

*Создано: 2026-03-07 | Статус: идея*

---

## Проблема

Когда агент выполняет shell-команду (execute_command), которая изменяет файлы, diff-система **не показывает эти изменения**. Файл меняется "молча" — без красного/зелёного diff-превью, без кнопок Accept/Reject.

Причина: diff-система перехватывает изменения только от инструментов `write_to_file` и `replace_in_file`, которые проходят через API расширения. Shell-команды меняют файлы напрямую в файловой системе, минуя diff-систему.

**Это же поведение у Cursor** — проверено экспериментально 2026-03-07. Shell-изменения не отображаются в diff-системе ни у Cursor, ни у Shuncode.

### Почему это важно

- Агент может через `sed`, `patch`, `npm init`, `git apply` и т.д. изменить десятки файлов
- Пользователь не видит что именно изменилось
- Нет возможности откатить отдельные изменения
- Безопасность: shell-команда может изменить файлы, о которых пользователь не знает

---

## Предлагаемое решение

### Концепция: Snapshot + Unified Diff + External Change Type

1. **До выполнения shell-команды** — `ExecuteCommandToolHandler` сохраняет snapshot файлов в рабочей директории (или отслеживаемых файлов)
2. **После выполнения** — сравнивает файлы со snapshot, находит изменённые
3. **Для каждого изменённого файла** — вычисляет unified diff и показывает в diff-системе
4. **Новый тип операции** — `external_modification` (не replace/delete/insert, а отдельный тип)

### Как отмечать изменения

В отличие от `write_to_file` / `replace_in_file`, где мы точно знаем что добавлено/удалено/заменено, для shell-изменений:

- Показываем **unified diff** целиком (как `git diff`)
- Тип: `external_modification` — визуально отличается (другой цвет/иконка)
- Accept = "ок, принято" (файл уже изменён, accept просто закрывает diff)
- Reject = `git checkout -- <file>` (откат через git, надёжнее чем backup)

### Ограничения и компромиссы

- **Snapshot всей директории** — дорого для больших проектов. Варианты оптимизации:
  - Использовать `git status` / `git diff` вместо собственных snapshot'ов
  - Следить только за файлами в пределах workspace
  - Использовать `fs.watch` / FileSystemWatcher во время выполнения команды
- **Бинарные файлы** — пропускаем, показываем только "файл изменён"
- **Новые файлы** — показываем как полный diff (весь контент зелёный)
- **Удалённые файлы** — показываем как полный diff (весь контент красный)

## Варианты реализации (от простого к сложному)

### Вариант А: git-based (минимальный, рекомендуемый для старта)

```
До команды:  git stash --keep-index --include-untracked (или git diff --stat)
После:       git diff --name-only → для каждого файла git diff <file>
```

Плюсы: просто, надёжно, работает с git.
Минусы: не работает вне git-репозитория.

### Вариант Б: FileSystemWatcher

Подписываемся на `vscode.workspace.onDidChangeTextDocument` и `FileSystemWatcher` на время выполнения команды. Собираем список изменённых файлов.

Плюсы: работает без git.
Минусы: может пропустить быстрые изменения, сложнее получить "что было до".

### Вариант В: Snapshot (полный)

Копируем содержимое файлов в память до команды, сравниваем после.

Плюсы: точный, работает без git.
Минусы: дорого по памяти для больших проектов. Нужна эвристика — какие файлы снапшотить.

## Оценка трудоёмкости

- Вариант А (git-based): ~6-10 часов
- Вариант Б (FileSystemWatcher): ~10-16 часов
- Вариант В (Snapshot): ~12-20 часов

## Открытые вопросы

1. Нужен ли snapshot для ВСЕХ shell-команд, или только для тех, что потенциально меняют файлы? (эвристика по команде: `echo`, `ls`, `git log` — не меняют)
2. Как быть с командами, которые работают долго (npm install, build)? Snapshot может устареть.
3. Стоит ли показывать diff для файлов в node_modules / .git?
4. Нужен ли откат для новых файлов (которых не было до команды)?
