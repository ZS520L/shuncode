# Управление контекстом (Context Management)

> Как Shuncode обрабатывает накопление истории чата, когда и почему происходит сжатие контекста, и какие улучшения планируются.

*Создано: 2026-03-07*

---

## Проблема

При длительной работе в чате история сообщений неизбежно растёт: каждый промпт пользователя, ответ модели, результаты tool calls (чтение файлов, команды, diff-превью) — всё это накапливается в `apiConversationHistory`. Модели имеют ограниченное контекстное окно (64k–200k токенов), и при его переполнении запрос просто упадёт с ошибкой.

Текущая система решает эту проблему **реактивно** — сжатие срабатывает только когда использование токенов приближается к лимиту. Это означает, что бо́льшую часть сессии модель работает с раздутым контекстом, что ведёт к:

- **Высокой стоимости** — каждый запрос включает всю историю, input tokens растут линейно.
- **Замедлению** — чем больше токенов на входе, тем дольше обработка.
- **Снижению качества** — модели хуже работают с очень длинным контекстом (lost-in-the-middle problem).

### Две стратегии сжатия

#### 1. Стандартная обрезка (Legacy Truncation)

Работает когда `useAutoCondense = false` или модель не входит в `isNextGenModelFamily`.

**Алгоритм:**
1. После каждого API-запроса проверяется `totalTokens >= maxAllowedSize`.
2. Если порог превышен — сначала пытается оптимизировать файловые чтения (`attemptFileReadOptimizationCore`): повторные чтения одного файла заменяются заглушкой.
3. Если оптимизация сэкономила < 30% контекста — выполняется обрезка: удаляются сообщения из середины истории (сохраняются первая пара user/assistant и последние сообщения).
4. В первое сообщение ассистента вставляется уведомление о том, что часть истории удалена.

**Расчёт `maxAllowedSize`** (буфер под генерацию ответа):

```
contextWindow  64k  → maxAllowed = 64k  - 27k = 37k
contextWindow 128k  → maxAllowed = 128k - 30k = 98k
contextWindow 200k  → maxAllowed = 200k - 40k = 160k
default             → maxAllowed = max(contextWindow - 40k, contextWindow * 0.8)
```

#### 2. Авто-конденсация (Auto-Condense / Summarization)

Работает когда `useAutoCondense = true` **И** модель входит в `isNextGenModelFamily` (Claude 4+, Gemini 2.5/3, GPT-5, Grok 4, DeepSeek 3.2, Minimax, и другие next-gen модели).

**Алгоритм:**
1. Перед каждым API-запросом проверяется `shouldCompactContextWindow()`:
   - `totalTokens >= min(contextWindow * autoCondenseThreshold, maxAllowedSize)`
2. Если порог превышен — сначала пытается оптимизировать файловые чтения.
3. Если оптимизация недостаточна — к сообщению пользователя добавляется промпт `summarizeTask`.
4. Модель генерирует структурированный отчёт через tool call `summarize_task`:
   - Основная задача и намерения пользователя
   - Технические решения и паттерны
   - Изменённые файлы и ключевой код
   - Нерешённые задачи и следующий шаг
   - Список "Required Files" для автоматического перечитывания
5. `SummarizeTaskHandler` удаляет **всю** старую историю (стратегия `"none"`), вставляет суммари как continuation prompt, и автоматически читает до 8 файлов из списка Required Files (до 100k символов суммарно).

**Настройка порога:**
- `autoCondenseThreshold` — число от 0 до 1, по умолчанию **0.75**.
- UI: клик по progress bar контекстного окна в шапке чата.
- Клавиши: ←/→ (шаг 5%), Shift+←/→ (шаг 10%).

### Обработка ошибок переполнения

Если несмотря на все оптимизации контекст всё равно переполняется, `context-error-handling.ts` детектирует ошибки от провайдеров (OpenAI, Anthropic, OpenRouter, Bedrock, Cerebras, Vercel) и позволяет системе корректно среагировать.

## Проблемы текущего подхода

### 1. Реактивность вместо проактивности
Сжатие срабатывает только при приближении к лимиту. До этого момента каждый запрос отправляет **всю** историю, включая давно неактуальные обсуждения. Это расточительно по токенам и стоимости.

### 2. `useAutoCondense` по умолчанию выключен
Большинство пользователей не знают о существовании авто-конденсации и работают с legacy truncation, которая теряет информацию без суммаризации.

### 3. Ограничение по моделям
Авто-конденсация работает только для `isNextGenModelFamily`. Модели вроде Claude 3.5, GPT-4o, Qwen, обычный DeepSeek — используют грубую обрезку.

### 4. Нет контроля «агрессивности» сжатия
Пользователь может выставить порог (25%–100%), но нет возможности задать периодическое сжатие (например, каждые N промптов) или гибридную стратегию.

---

## Предлагаемое решение: Гибкое управление контекстом

### Концепция

Добавить два взаимодополняющих механизма, которые пользователь может комбинировать:

1. **Порог по токенам** (уже есть) — `autoCondenseThreshold`, диапазон 0.25–1.0
2. **Интервал по промптам** (новый) — `autoCondenseInterval`, сжатие каждые N пользовательских сообщений

### Новая настройка: `autoCondenseInterval`

**Где добавить:**

```
src/shared/storage/state-keys.ts  →  USER_SETTINGS_FIELDS
```

```typescript
autoCondenseInterval: { default: 0 as number },
// 0 = отключено (по умолчанию)
// N > 0 = сжимать каждые N пользовательских промптов
```

**Логика принятия решения** (в `recursivelyMakeShuncodeRequests`):

```
shouldCompact = false

// Стратегия 1: По токенам (существующая)
if (totalTokens >= threshold) {
    shouldCompact = true
}

// Стратегия 2: По интервалу (новая)
if (interval > 0 && apiRequestCount % interval === 0 && apiRequestCount > 0) {
    shouldCompact = true
}
```

### Рекомендуемые пресеты

| Пресет | Threshold | Interval | Для кого |
|---|---|---|---|
| **Экономный** | 0.30 | 5 | Дешёвые модели, простые задачи |
| **Сбалансированный** | 0.50 | 0 | Основной режим для большинства |
| **По умолчанию** | 0.75 | 0 | Текущее поведение, без изменений |
| **Максимум памяти** | 1.0 | 0 | Дорогие модели с большим окном |

### Снятие ограничения по моделям

Авто-конденсация (`summarize_task`) требует, чтобы модель умела делать tool calls и генерировать качественный отчёт. Ограничение `isNextGenModelFamily` слишком жёсткое — многие модели (Claude 3.5 Sonnet, GPT-4o, Qwen 2.5 72B) вполне способны это делать.

**Предложение:** Заменить проверку `isNextGenModelFamily` на проверку минимального размера контекстного окна (например, >= 32k токенов). Модели с маленьким окном (< 32k) продолжат использовать legacy truncation.

### UI изменения

В секции настроек **Контекст** (`settings.tabs.context`) добавить:

1. **Слайдер «Порог сжатия»** — от 25% до 100% (уже есть в progress bar, но стоит продублировать в настройках для наглядности).
2. **Поле «Интервал сжатия»** — число (0 = отключено, 3–10 = рекомендуемый диапазон).
3. **Тултип с объяснением** — что делает каждая настройка и как они взаимодействуют.

## Компромиссы

### Частое сжатие (низкий порог / малый интервал)
- ✅ Дешевле каждый отдельный запрос (меньше input tokens)
- ✅ Модель работает с «чистым» контекстом
- ❌ Каждая суммаризация — отдельный запрос к LLM (время + деньги)
- ❌ Риск потери деталей при суммаризации (drift)
- ❌ Прерывает flow работы на 5–15 секунд

### Редкое сжатие (высокий порог / без интервала)
- ✅ Модель видит полную историю — меньше ошибок контекста
- ✅ Нет задержек на суммаризацию
- ❌ Дорого по input tokens
- ❌ lost-in-the-middle: модель хуже «видит» информацию из середины длинного контекста
- ❌ Если сжатие всё-таки нужно — оно будет масштабным и может потерять больше

### Счётчик промптов vs. процент токенов
Счётчик промптов — плохая метрика сама по себе: один промпт с вставленным логом на 10k строк весит столько же, сколько 50 обычных сообщений. **Поэтому интервал по промптам должен работать как дополнение к порогу по токенам, а не как замена.**

## План реализации

### Фаза 1: Минимальные изменения (1–2 часа)
- [ ] Включить `useAutoCondense: true` по умолчанию
- [ ] Снизить `autoCondenseThreshold` до 0.50
- [ ] Убрать или ослабить проверку `isNextGenModelFamily` (заменить на проверку contextWindow >= 32k)

### Фаза 2: Интервал по промптам (2–4 часа)
- [ ] Добавить `autoCondenseInterval` в `state-keys.ts`
- [ ] Обновить proto (`state.proto`)
- [ ] Добавить логику проверки интервала в `recursivelyMakeShuncodeRequests`
- [ ] Добавить UI-контрол в настройки (секция «Контекст»)
- [ ] i18n строки (RU + EN)

### Фаза 3: UX-улучшения (2–4 часа)
- [ ] Пресеты «Экономный / Сбалансированный / Максимум памяти» в настройках
- [ ] Уведомление пользователю при срабатывании сжатия (toast / chat message)
- [ ] Статистика: сколько токенов сэкономлено за сессию
