# IMPL-18: Биллинг и лицензирование

> Статус: ❌ Не реализовано
> Приоритет: Средний (нужен после успешного пилота)
> Оценка: 10–15 дней
> Зависимости: нет

---

## 1. Цель

Реализовать систему управления лицензиями и биллинга для корпоративных клиентов. Текущая система (ручная генерация ключей в админке, проверка через `/api/license/verify`) покрывает пилот, но не масштабируется на десятки клиентов и сотни пользователей.

## 2. Текущее состояние

### Что есть
- **Модель License** в Prisma: `key`, `plan` (free/pro), `status` (active/revoked/expired), `expiresAt`, `userId`
- **Админка** (`/admin/licenses`): ручная генерация ключа, отзыв, восстановление
- **API** `POST /api/license/verify`: расширение проверяет ключ при запуске
- **Профиль** `/api/v1/users/me`: возвращает массив лицензий пользователя

### Чего нет
- Тарифные планы с ограничениями (лимиты запросов, фичи)
- Корпоративные лицензии (N мест на организацию)
- Автоматическое истечение и продление
- Организации (группировка пользователей по компаниям)
- Трекинг использования (сколько запросов, кто активен)
- Оплата (приём платежей)
- Self-service: клиент сам покупает/продлевает

## 3. Архитектура

### 3.1. Модель данных (Prisma)

```prisma
model Organization {
  id          String   @id @default(cuid())
  name        String
  slug        String   @unique
  plan        OrgPlan  @default(trial)
  maxSeats    Int      @default(5)
  createdAt   DateTime @default(now())
  expiresAt   DateTime?
  
  members     OrgMember[]
  invoices    Invoice[]
}

model OrgMember {
  id             String       @id @default(cuid())
  role           OrgRole      @default(member)
  joinedAt       DateTime     @default(now())
  
  userId         String
  user           User         @relation(fields: [userId], references: [id])
  organizationId String
  organization   Organization @relation(fields: [organizationId], references: [id])
  
  @@unique([userId, organizationId])
}

model UsageLog {
  id        String   @id @default(cuid())
  userId    String
  action    String   // "chat_message", "inline_edit", "search"
  tokens    Int?     // кол-во токенов (если применимо)
  model     String?  // какая модель использовалась
  createdAt DateTime @default(now())
  
  user      User     @relation(fields: [userId], references: [id])
}

model Invoice {
  id             String        @id @default(cuid())
  organizationId String
  organization   Organization  @relation(fields: [organizationId], references: [id])
  amount         Int           // в копейках
  currency       String        @default("RUB")
  status         InvoiceStatus @default(pending)
  periodStart    DateTime
  periodEnd      DateTime
  paidAt         DateTime?
  createdAt      DateTime      @default(now())
}

enum OrgPlan {
  trial       // бесплатный пилот, до 3 мес
  starter     // до 5 мест, 500 ₽/мес за место
  business    // до 50 мест, 400 ₽/мес за место (скидка за объём)
  enterprise  // без лимита, индивидуальные условия
}

enum OrgRole {
  owner       // создатель, полный доступ
  admin       // управление членами
  member      // обычный пользователь
}

enum InvoiceStatus {
  pending
  paid
  overdue
  cancelled
}
```

### 3.2. Связь с текущей системой

Текущая модель `License` остаётся для индивидуальных пользователей (free/pro). Организации — отдельная сущность. Пользователь может быть и индивидуальным, и членом организации.

Проверка доступа (в расширении):
1. `POST /api/license/verify` — проверяет индивидуальный ключ (как сейчас)
2. **Новый** `GET /api/v1/users/me` — возвращает `organizations[]` с планом и статусом
3. Расширение проверяет: есть ли активная индивидуальная лицензия ИЛИ активная организация

## 4. Фазы реализации

### Фаза 1: Организации (3–5 дней)

**Цель:** корпоративный клиент может объединить пользователей в организацию.

**Шаги:**

1. **Prisma:** добавить модели `Organization`, `OrgMember`
2. **API:**
   - `POST /api/v1/orgs` — создать организацию
   - `GET /api/v1/orgs/:id` — информация об организации
   - `POST /api/v1/orgs/:id/members` — добавить пользователя
   - `DELETE /api/v1/orgs/:id/members/:userId` — удалить
   - `PATCH /api/v1/orgs/:id/members/:userId` — сменить роль
3. **Админка:** новый раздел `/admin/organizations`
   - Таблица организаций: название, план, мест занято/макс, дата истечения
   - Создание организации (название, план, maxSeats, expiresAt)
   - Управление членами
4. **`GET /api/v1/users/me`:** расширить ответ — добавить `organizations[]`
5. **Расширение:** обновить проверку лицензии — учитывать членство в организации

**Результат:** админ создаёт организацию «Компания X», добавляет 5 пользователей, они получают доступ.

---

### Фаза 2: Трекинг использования (2–3 дня)

**Цель:** видеть, кто и сколько пользуется продуктом.

**Шаги:**

1. **Prisma:** добавить модель `UsageLog`
2. **API:** `POST /api/v1/usage/log` — расширение отправляет события
3. **Расширение:** отправлять события:
   - `chat_message` — отправка сообщения в чат
   - `inline_edit` — использование Ctrl+K
   - `search` — семантический поиск
   - Отправлять батчами (раз в 5 минут или при закрытии)
4. **Админка:** дашборд с графиками:
   - Активные пользователи за день/неделю/месяц
   - Количество запросов по типам
   - Топ пользователей по активности
5. **Отчёт для клиента:** экспорт в CSV / PDF (для обоснования продления)

**Результат:** в админке видно: 15 из 20 пользователей активны, 340 запросов за неделю.

**Важно:** трекинг отправляется на shuncode-web сервер. В enterprise on-premise сборке — трекинг опционален или отключён (данные не покидают контур). Нужен флаг конфигурации.

---

### Фаза 3: Автоматическое управление лицензиями (2–3 дня)

**Цель:** лицензии истекают и продлеваются автоматически.

**Шаги:**

1. **Cron-задача** (или Next.js API route с внешним cron):
   - Ежедневно проверять `expiresAt` организаций и индивидуальных лицензий
   - За 7 дней до истечения — email-уведомление владельцу
   - За 1 день — повторное уведомление
   - При истечении — статус `expired`, расширение показывает баннер
2. **Баннеры:** использовать существующую систему баннеров для уведомлений в расширении
3. **Grace period:** 7 дней после истечения — продукт работает, но с баннером

**Результат:** клиенту приходит email «Лицензия истекает через 7 дней», после истечения — баннер в IDE.

---

### Фаза 4: Оплата (3–5 дней)

**Цель:** клиент платит за лицензию через сайт.

**Варианты платёжных систем (Россия):**

| Система | Комиссия | Для юрлиц | Рекуррентные |
|---------|----------|-----------|-------------|
| ЮKassa | 3.5% | Да | Да |
| Тинькофф Оплата | 2.49% | Да | Да |
| CloudPayments | 2.5–3.5% | Да | Да |
| Stripe (через посредника) | 3.4% | Сложно из РФ | Да |

**Рекомендация:** ЮKassa или Тинькофф Оплата — оба поддерживают приём от юрлиц, рекуррентные платежи, и просты в интеграции.

**Шаги:**

1. **Зарегистрировать ООО** в платёжной системе (ЮKassa/Тинькофф)
2. **API:**
   - `POST /api/v1/billing/checkout` — создать платёж
   - `POST /api/v1/billing/webhook` — обработать callback от платёжной системы
   - `GET /api/v1/billing/invoices` — история платежей
3. **Prisma:** модель `Invoice`
4. **Страница на сайте:** `/pricing` — тарифы, кнопка «Купить»
5. **Страница в ЛК:** `/account/billing` — текущий план, история, кнопка «Продлить»

**Результат:** клиент выбирает тариф → платит картой → лицензия активируется автоматически.

---

### Фаза 5: Self-service портал (3–5 дней)

**Цель:** клиент сам управляет организацией, без обращения к админу.

**Шаги:**

1. **Страница** `/account/organization`:
   - Информация об организации (название, план, мест занято/макс)
   - Управление членами (добавить по email, удалить, сменить роль)
   - Invite-ссылки (как в Slack — ссылка для самостоятельного присоединения)
2. **Страница** `/account/billing`:
   - Текущий план и дата следующего платежа
   - Изменение количества мест
   - История счетов, скачивание актов

**Результат:** IT-директор клиента сам добавляет разработчиков, не обращаясь к нам.

## 6. Изменения в расширении (Shuncode IDE)

### Проверка лицензии (обновить)

Текущий поток:
1. Расширение при запуске вызывает `POST /api/license/verify` с ключом
2. Получает `plan` и `status`

Новый поток:
1. Расширение при запуске вызывает `GET /api/v1/users/me`
2. Ответ содержит `licenses[]` и `organizations[]`
3. Доступ есть, если:
   - Есть активная индивидуальная лицензия (plan: free/pro, status: active)
   - ИЛИ пользователь — member организации с активным планом и не истёкшим сроком
4. Если ни одного — показать баннер «Требуется лицензия»

### Трекинг использования (новый)

- Собирать события локально в массив
- Отправлять батчем на `POST /api/v1/usage/log` каждые 5 минут
- При закрытии IDE — отправить оставшиеся
- **Конфигурация:** флаг `shuncode.telemetry.usage` (default: true для cloud, false для on-premise)
- Не блокировать работу при ошибках отправки

## 7. Приоритеты

| Фаза | Что | Когда нужно |
|------|-----|-------------|
| 1 | Организации | После первого корпоративного клиента (>5 человек) |
| 2 | Трекинг | Вместе с Фазой 1 (нужен для отчётов клиенту) |
| 3 | Автоматические лицензии | После 3+ клиентов |
| 4 | Оплата | После 5+ клиентов (до этого — счета вручную) |
| 5 | Self-service | После 10+ клиентов |

**Для текущего пилота (1 клиент, 3–7 человек) достаточно того, что есть:** ручная генерация ключей в админке. Фазы 1–5 — по мере роста.
